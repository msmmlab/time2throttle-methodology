# Time2Throttle — Test Methodology

**Version 0.1 — draft, open for critique** · June 2026
Operated by [mmlab Pty Ltd](https://www.mmlab.ie), Brisbane, Australia · Published at [time2throttle.com/methodology](https://time2throttle.com/methodology)
Status: **no results have been published under this methodology yet — by design.** The methodology precedes the numbers so the numbers can be trusted.

> This repository is the source of truth for the Time2Throttle test methodology. The website renders this file. Changes happen only here, by versioned revision with changelog — never retroactively. Found a flaw? [Open an issue](../../issues) — credited corrections enter the changelog.

## 1. Purpose

An independent, reproducible methodology for measuring thermal and battery behaviour of smart glasses and head-mounted wearables under realistic data workloads.

Vendor runtime claims diverge consistently from real-world performance, and no standardized, independent, published measurement exists for this category. Phones have independent reviewers; cameras have DxOMark; TVs have RTINGS; smart glasses have marketing copy. This document defines how Time2Throttle produces its numbers — before the first number is produced.

**Reproducibility is the product.** The workload app, raw telemetry archives, and this methodology are published so anyone can re-run any test and check any result.

## 2. Scope & device tiers

| Tier | Devices | What we can measure |
|---|---|---|
| **Instrumented** | Android-based devices (most enterprise HMDs; Android-based consumer AR) — profiler app sideloaded | Full telemetry: thermal headroom & status, power draw, pipeline metrics, plus external surface temperature |
| **Black-box** | Closed ecosystems that allow no third-party installs | External-only: standardized-usage runtime, surface temperature, charge analysis, claim delta |
| **Reference** | Phone-class Android 11+ devices | Methodology development & calibration only — phone results are never published as wearable findings |

Out of scope in v0.1: VR headsets (different thermal envelope and mounting), audio-only frames without camera/display load.

## 3. Test environment

- **Standard ambient:** 23 °C ± 1 °C, still air. All published scores at standard ambient.
- **Stress ambient:** 40 °C ± 1 °C. Reported separately; never averaged with standard-ambient results.
- **Mounting:** worn-equivalent — head-form or insulated stand replicating skin-adjacent dissipation. Open-desk bench testing is invalid: it overstates dissipation and flatters every device.
- **Device state:** battery 100% at start; fixed, declared display brightness; charging disabled; radios active — the radio is part of the workload; vendor thermal modes at factory default, with any deviation documented in the result.
- **Runs:** minimum 2 per workload per configuration; the median is reported; divergence above 10% on any primary metric is flagged in the published result.

## 4. Instrumentation

**Internal (instrumented tier):**
- Thermal: headroom sampled at 1 Hz plus thermal-status transition events (Android Thermal API).
- Power: instantaneous current draw at 1 Hz and charge-counter delta per run (BatteryManager).
- Pipeline: encoder bitrate/resolution/fps actuals and bytes transmitted.
- All telemetry is timestamped at source; device–server clock offset is measured at session start.
- Devices below modern API levels: battery temperature and readable system thermal zones, with per-device capability documented in the result.

**External (all tiers):**
- Surface temperature via thermal camera or contact probes at **≥3 declared skin-contact points** (temple arms, nose bridge / forehead pad).
- **Surface temperature is the primary comfort metric.** Internal sensors alone do not constitute a valid result: the skin-contact ceiling (~39 °C, comfort/safety convention) is measured where skin actually touches, not where the SoC reports.

## 5. Workload profiles

All workloads are generated by the open-source Time2Throttle workload app — versioned, identical APK across instrumented devices. Black-box devices run a scripted, documented standardized-usage equivalent.

| ID | Name | Definition | Simulates |
|---|---|---|---|
| W0 | Idle baseline | Display on, no capture, no upload, 30 min | Attribution floor |
| W1 | Sustained uplink | Camera capture → hardware encode (declared settings) → continuous upload, until throttle event or battery ≤ 10% | Remote-assist / streaming — the canonical heavy use |
| W2 | Burst | W1 load in 90 s bursts, 5 min gaps, 10 cycles | Inspection / evidence-capture work |
| W3 | Display-heavy | Max-brightness rendering loop, no capture | Display attribution |

**Configuration matrix (run on W1 at minimum):**
- **Config A** — full quality: the device/app's declared maximum sustainable settings.
- **Config B** — reduced pipeline: ~50% bitrate, reduced fps/resolution (declared).
- The A/B delta quantifies how much of a device's thermal and battery problem is attributable to the data pipeline versus fixed loads (display, sensors, SoC baseline).

## 6. Metrics & definitions

| Metric | Definition |
|---|---|
| **Time-to-Throttle (T2T)** | Minutes from workload start to the first performance limit — **with the cause recorded: thermal-triggered or power-triggered.** Heat and degraded batteries both end the session; the cause column says which one did. |
| **Surface plateau** | Maximum stabilized surface temperature at the hottest declared contact point (°C). |
| **Real runtime** | Minutes from 100% to 10% battery under workload. The 10% floor prevents deep-discharge damage so retail units can be accurately re-tested. |
| **Claim delta** | Measured runtime vs vendor-claimed runtime (%). Reported for every device — the headline number. |
| **Pipeline slice** | Share of average power draw attributable to encode + radio + pipeline processing, derived from W0 / W1 / W3 decomposition. |
| **Config delta** | Improvement of Config B over Config A: Δ time-to-throttle and Δ runtime (%). |
| **Recovery time** | Minutes from a throttle event back to the unthrottled state at idle. |

## 7. Reporting & publication rules

- Every published result includes: device identity, firmware version, methodology version, full configuration matrix, downloadable raw telemetry archive, and thermal images of the contact points.
- Headline format per device: **Claim delta · T2T (with cause) · Surface plateau · Real runtime** — measured on W1, Config A, standard ambient.
- **Negative and boring results are published.** No tested device is exempt once testing begins.
- **Errata are append-only.** Results are timestamped and hash-anchored at publication; corrections are appended with dates, and history is never silently edited.
- Composite scoring is deliberately deferred to a later methodology version: premature single-number scores invite disputes before the per-metric data has earned trust.

## 8. Independence rules

1. Scores and results are **never** purchasable, sponsorable, or removable.
2. Test devices are bought at retail. Vendor-supplied units are disclosed inline with that device's results.
3. Paid services (e.g., pre-release testing) run under this exact public methodology, current version. A vendor may choose **whether** a pre-release result is published — never **what** it says.
4. Affiliate links, where used, are added after the verdict and never influence it.
5. Any commercial relationship with a measured vendor is disclosed inline with that vendor's results.
6. This methodology changes only by public versioned revision with changelog — never retroactively.

## 9. Critique invited

This is v0.1, published before any result on purpose. If you build, deploy, test, or wear these devices and you see a flaw — in the workloads, the mounting, the statistics, the metric definitions — [open an issue](../../issues). Substantive corrections are incorporated by versioned revision and credited in the changelog.
