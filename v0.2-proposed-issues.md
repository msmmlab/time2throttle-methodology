## Issue 1 — Control humidity in test environment

**Title:** Add humidity as a controlled variable in Section 3 test environment  
**Labels:** `methodology` `v0.2` `test-environment`

### Summary
Section 3 currently defines ambient temperature but does not define humidity.
Humidity should be controlled and declared because it affects both thermal
dissipation and perceived skin-contact comfort.

### Proposed change
Update Section 3 to include:

- Standard ambient: **23 °C ± 1 °C, 50% ± 5% RH, still air**
- Stress ambient: **40 °C ± 1 °C, 50% ± 5% RH, still air**

If the same RH target is not desired for stress ambient, document the
alternative explicitly.

### Rationale
Humidity influences:
- thermal behaviour at skin-contact surfaces
- perceived comfort at a given surface temperature
- reproducibility across test sites in different climates

### Acceptance criteria
- [ ] Section 3 includes RH in standard ambient definition
- [ ] Section 3 includes RH in stress ambient definition
- [ ] Formatting matches existing methodology style
- [ ] Change is recorded in changelog for v0.2

## Issue 2 — Increase minimum runs from 2 to 3

**Title:** Raise minimum run count to 3 so median reporting is statistically valid  
**Labels:** `methodology` `v0.2` `statistics`

### Summary
Section 3 currently requires a minimum of 2 runs per workload per
configuration while stating that the median is reported. With only 2 runs,
a true median does not exist in the intended sense.

### Proposed change
Change the requirement from:
- minimum **2** runs per workload per configuration

to:
- minimum **3** runs per workload per configuration

Retain:
- reporting the **median**
- flagging divergence above **10%** on any primary metric

### Rationale
Three runs:
- make median reporting statistically valid
- reduce sensitivity to one-off run anomalies
- improve credibility of published results without significantly increasing
  test burden

### Acceptance criteria
- [ ] Section 3 updates minimum runs from 2 to 3
- [ ] Median language remains and is now statistically coherent
- [ ] 10% divergence rule is preserved
- [ ] Changelog includes this revision in v0.2

## Issue 3 — Specify controlled network conditions for W1

**Title:** Define controlled network conditions for W1 sustained uplink workload  
**Labels:** `methodology` `v0.2` `workloads` `networking`

### Summary
W1 depends on continuous upload, but the methodology does not yet define
network conditions. Uncontrolled latency or bandwidth can alter encoder
behaviour, buffering, radio load, and thermal outcomes — making results
unreliable across test sites.

### Proposed change
Add to Section 5 / W1 definition that tests are conducted under controlled
local network conditions:

- minimum **100 Mbps available bandwidth**
- maximum **5 ms latency**

Add a note clarifying that this removes network congestion as a variable
in thermal and battery outcomes.

### Rationale
Without network control, W1 results may reflect:
- network instability at the test site
- uplink congestion unrelated to the device
- transport-layer backpressure affecting encoder behaviour

rather than the device's own thermal and power characteristics.

### Acceptance criteria
- [ ] W1 includes network-condition requirements
- [ ] Minimum bandwidth is stated as 100 Mbps
- [ ] Maximum latency is stated as 5 ms
- [ ] Text explains why this control exists
- [ ] Changelog includes this revision in v0.2

## Issue 4 — Tighten Recovery Time definition

**Title:** Replace vague Recovery Time definition with precise thermal + surface criteria  
**Labels:** `methodology` `v0.2` `metrics` `thermal`

### Summary
Section 6 currently defines Recovery Time as:

> "Minutes from a throttle event back to the unthrottled state at idle."

This is too vague for consistent cross-device comparison. "Unthrottled state"
is undefined and may be interpreted differently across test operators or devices.

### Proposed change
Replace the current definition with:

Recovery Time = minutes from a throttle event until **both** of the
following conditions are true:

1. Thermal status has returned to **NONE** per Android Thermal API
2. Surface temperature at the hottest declared contact point has dropped
   below **37 °C**

The recorded recovery endpoint is whichever of the two conditions occurs
**last**.

Add a note explaining that this dual-condition definition confirms recovery
at both:
- internal SoC / system level
- skin-contact comfort level

### Rationale
A device may recover internally before it is comfortable to wear again,
or vice versa. Both conditions matter for a wearable benchmark. Single-
condition definitions risk flattering devices that cool internally but
remain hot at the skin-contact surface.

### Acceptance criteria
- [ ] Recovery Time definition is replaced in Section 6
- [ ] Android Thermal API NONE criterion is included
- [ ] Surface temperature < 37 °C criterion is included
- [ ] "Whichever occurs last" rule is included
- [ ] Explanatory note is added linking internal and comfort recovery
- [ ] Changelog includes this revision in v0.2

## Issue 5 — Add limitation note for Pipeline Slice decomposition

**Title:** Document linearity limitation in Pipeline Slice power decomposition  
**Labels:** `methodology` `v0.2` `metrics` `limitations`

### Summary
Section 6 defines Pipeline Slice using W0 / W1 / W3 decomposition. This
is a useful approach, but it implicitly assumes linear and independent
contribution of components to total power draw — an assumption that does
not always hold in practice.

### Proposed change
Add a known limitation note under Pipeline Slice in Section 6 stating that:

- the decomposition assumes linear and independent contribution of
  components to power draw
- non-linear interactions between components are a known limitation
  of this approach
- a key example is thermal throttling during W1 suppressing CPU/GPU
  loads that appear at their full level when isolated in W0 or W3
- this limitation will be revisited and addressed in a future
  methodology version

### Rationale
Explicitly acknowledging model assumptions makes the methodology more
defensible under scrutiny, prevents over-interpretation of Pipeline Slice
values in cross-device comparison, and preserves trust before composite
scoring is introduced in a later version.

### Acceptance criteria
- [ ] Pipeline Slice metric includes an explicit limitation note in Section 6
- [ ] Linearity and independence assumption is stated
- [ ] Non-linear thermal throttling interaction is named as an example
- [ ] Future revision commitment is noted
- [ ] Changelog includes this revision in v0.2

## Issue 6 — Disclose reproducibility limits for black-box tier

**Title:** Explicitly disclose third-party reproducibility limits for black-box tier  
**Labels:** `methodology` `v0.2` `scope` `disclosure`

### Summary
Section 2 describes the Black-box tier honestly in terms of what can be
measured. However, it does not explicitly state that these results cannot
be fully reproduced by third parties due to the absence of internal
telemetry access — which creates a tension with the methodology's founding
principle that reproducibility is the product.

### Proposed change
Add an inline note or footnote to the Black-box tier row in Section 2
stating that:

- black-box results cannot be fully reproduced by third parties because
  internal telemetry is unavailable for this device class
- this reproducibility limitation is explicitly disclosed inline with
  every published black-box result
- external measurements (runtime, surface temperature, charge analysis)
  remain independently verifiable

### Rationale
Where full reproducibility is structurally impossible, the limitation
should be stated upfront in the tier definition — not deferred to the
individual result page. This is consistent with the methodology's
commitment to transparency before numbers are published.

### Acceptance criteria
- [ ] Section 2 Black-box tier row includes reproducibility disclosure
- [ ] Disclosure names the reason: no internal telemetry access
- [ ] Disclosure confirms the limitation will appear inline with all
  published black-box results
- [ ] Disclosure notes which external measurements remain verifiable
- [ ] Changelog includes this revision in v0.2
